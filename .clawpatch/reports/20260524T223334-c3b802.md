## medium: Cross-repo commit history can miss the newest commits when one repo has more than 10 recent commits

id: fnd_sig-feat-library-fd49585146-130b_0337d9caa1
category: bug
confidence: high
triage: confirmed-bug
status: open
feature: Node source src/lib (feat_library_fd49585146)

evidence:
- src/lib/data/index.ts:79-80
- src/lib/data/github.ts:178-194
- src/lib/data/github.ts:252-254

loadSignalData asks for 12 commits, and fetchAllRepoCommits advertises returning the newest limit after merging histories, but the GraphQL query only fetches history(first: 10) for each repo. If the most active repo has the 11th and 12th newest commits overall, those commits are never fetched, so the final sorted slice can backfill older commits from other repos or return fewer than requested. This makes the portfolio activity feed stale or misordered for common single-repo work bursts.

recommendation:
Request at least history(first: limit) per repo, and consider paginating repositories and histories if the UI contract is truly cross-repo newest commits. Keep the final merge-and-sort cap after fetching enough candidates.

test analysis:
The included tests cover local git parsing, shift clustering, and Substack parsing only. There is no test for fetchAllRepoCommits or its GraphQL pagination limits.

suggested regression test:
Add a Vitest test that stubs global fetch for the GraphQL endpoint, simulates one repo with more than 10 newest commits, calls fetchAllRepoCommits(12), and asserts the 11th and 12th newest commits are included after the fix.

minimum fix scope:
Update fetchAllRepoCommits query sizing and add focused tests for the cross-repo merge limit behavior.

repro:
Mock the GraphQL response so repo A has 12 commits newer than repo B's commits, but expose only the first 10 because the query asks history(first: 10). fetchAllRepoCommits(12) cannot include repo A's 11th and 12th commits and will return older repo B commits instead.

## medium: Git tests depend on mutable local repository history

id: fnd_sig-feat-test-suite-aaab826675-b_76545ccd9b
category: test-gap
confidence: high
triage: test-gap
status: open
feature: Test Suite (feat_test-suite_aaab826675)

evidence:
- tests/lib/data/git.test.ts:122-136
- src/lib/data/git.ts:98-105

The test suite calls the real git executable against the current checkout and then asserts properties of whatever commits happen to exist, including requiring at least one changed commit in the last 30 days. That makes npm run test depend on wall-clock time and clone history rather than only on the parser contract. A correct implementation can fail once the repo has no qualifying recent commits, in shallow/non-git distributions, or in CI setups with unusual checkout depth.

recommendation:
Keep parser behavior covered with deterministic fixture strings and move real-repo smoke coverage behind a guarded integration test that skips when git history is unavailable or insufficient. If the smoke test stays, remove the fixed 30-day assumption and assert only that the function handles the observed repository state without throwing.

test analysis:
These are the tests themselves; they validate against the live repository instead of pinning the expected git output fixture, so they cannot protect against environment-dependent failures.

suggested regression test:
Add a deterministic getCommitHistory-style parser fixture for a shortstat block, and add a guarded integration smoke test that skips when git log cannot provide at least one commit.

minimum fix scope:
Update tests/lib/data/git.test.ts to remove or guard live-repo history assumptions.

repro:
Run the test suite from a checkout with no commits in the last 30 days, from an exported source archive without .git, or with git history filtered such that getCommitHistory(30) returns no changed commits.

## medium: llms.txt analytics logs unsanitized request headers

id: fnd_sig-feat-route-8457fdd2c3-ba438e_a497d11b6d
category: security
confidence: high
triage: confirmed-bug
status: open
feature: SEO And Discoverability (feat_route_8457fdd2c3)

evidence:
- src/pages/llms.txt.ts:16-28 (GET)
- src/pages/llms.txt.ts:33-45 (GET)

The public /llms.txt route reads user-controlled user-agent and referer headers, logs them verbatim to Vercel function logs, and also forwards referer into analytics with only length truncation. Referer values can contain query strings, tokens, email addresses, or other sensitive data from upstream pages, and user-agent can be arbitrarily long or identifying. This creates an avoidable privacy/logging exposure on a public endpoint whose purpose is discoverability, not request forensics.

recommendation:
Sanitize before logging or tracking: drop referer query strings and fragments, record only origin or hostname, cap every logged field, and consider hashing or omitting user-agent. Keep raw headers out of console logs and analytics payloads.

test analysis:
No linked tests are included for the route, and there is no evidence of a test asserting that analytics/log payloads strip sensitive referer data or bound all user-controlled header values.

suggested regression test:
Add a route-level test that sends Referer with query/fragment secrets and an oversized User-Agent, stubs console.log and track, and asserts only sanitized bounded values are emitted while the response body remains the static llms.txt content.

minimum fix scope:
Update src/pages/llms.txt.ts to sanitize and bound telemetry fields before both console logging and track().

repro:
Request /llms.txt with a Referer such as https://example.com/callback?token=secret-value and a long User-Agent; the current handler records those values in function logs and sends the first 120 referer characters to analytics.

## medium: Archive workflow moves user-supplied change paths without validation or quoting

id: fnd_sig-feat-agent-tool-5410120634-9_30955b9b9d

triage: risk
status: open

- .claude/skills/openspec-archive-change/SKILL.md:75-83
- .codex/skills/openspec-archive-change/SKILL.md:75-83

The archive instructions build filesystem paths directly from the supplied change name and show an unquoted mv command. Because the workflow accepts an optional user-provided change name, a malicious or malformed name containing shell metacharacters, spaces, globs, or path traversal could cause the agent to execute an unintended shell operation or move the wrong directory. test analysis:
No tests are included for these workflow assets, and the issue is in agent command/skill documentation rather than executable application code.

suggested regression test:
Add a lightweight documentation or workflow validation test that scans OpenSpec archive instructions for safe quoting and explicit change-name validation before mv/process-exec examples.

minimum fix scope:
Update the Claude and Codex archive command/skill docs to add validation requirements and replace the raw mv examples with shell-safe quoted forms.
