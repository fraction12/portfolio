## medium: Cross-repo commit history can miss the newest commits when one repo has more than 10 recent commits

id: fnd_sig-feat-library-fd49585146-130b_0337d9caa1
category: bug
confidence: high
triage: confirmed-bug
status: open
feature: Node source src/lib (feat_library_fd49585146)

evidence:
- src/lib/data/index.ts:79-80
- src/lib/data/github.ts:178-194
- src/lib/data/github.ts:252-254

loadSignalData asks for 12 commits, and fetchAllRepoCommits advertises returning the newest limit after merging histories, but the GraphQL query only fetches history(first: 10) for each repo. If the most active repo has the 11th and 12th newest commits overall, those commits are never fetched, so the final sorted slice can backfill older commits from other repos or return fewer than requested. This makes the portfolio activity feed stale or misordered for common single-repo work bursts.

recommendation:
Request at least history(first: limit) per repo, and consider paginating repositories and histories if the UI contract is truly cross-repo newest commits. Keep the final merge-and-sort cap after fetching enough candidates.

test analysis:
The included tests cover local git parsing, shift clustering, and Substack parsing only. There is no test for fetchAllRepoCommits or its GraphQL pagination limits.

suggested regression test:
Add a Vitest test that stubs global fetch for the GraphQL endpoint, simulates one repo with more than 10 newest commits, calls fetchAllRepoCommits(12), and asserts the 11th and 12th newest commits are included after the fix.

minimum fix scope:
Update fetchAllRepoCommits query sizing and add focused tests for the cross-repo merge limit behavior.

repro:
Mock the GraphQL response so repo A has 12 commits newer than repo B's commits, but expose only the first 10 because the query asks history(first: 10). fetchAllRepoCommits(12) cannot include repo A's 11th and 12th commits and will return older repo B commits instead.

## medium: GitHub GraphQL errors can overwrite a good snapshot with an empty heatmap

id: fnd_sig-feat-library-fd49585146-83a2_18e9456a15
category: data-loss
confidence: high
triage: confirmed-bug
status: open
feature: Node source src/lib (feat_library_fd49585146)

evidence:
- src/lib/data/github.ts:30-39
- src/lib/data/github.ts:75-84

fetchHeatmap treats any HTTP 200 response as usable data and defaults missing contribution data to an empty array. GitHub GraphQL can return HTTP 200 with an errors payload and null or partial data. In that case loadGithub will still build a fresh snapshot with commits7d and commits30d as 0, heatmap as [], and then write it to src/data/snapshots/github.json, replacing the last known-good snapshot instead of falling back to it.

recommendation:
After parsing the GraphQL JSON, explicitly check json.errors and validate the expected weeks array before returning data. Throw on errors or malformed contribution data so loadGithub uses readSnapshot and does not write corrupted fresh data.

test analysis:
No included tests exercise loadGithub or fetchHeatmap, and none simulate GraphQL error payloads with successful HTTP status codes.

suggested regression test:
Add a loadGithub test with global fetch mocked so the GraphQL call returns a 200 error payload, then assert that the snapshot fallback is used and no empty heatmap snapshot is written.

minimum fix scope:
Add GraphQL response validation in fetchHeatmap and a regression test for 200-with-errors fallback behavior.

repro:
Stub the repos request to succeed and the GraphQL request to return HTTP 200 with {"errors":[{"message":"rate limited"}],"data":null}. loadGithub will construct and write a snapshot with an empty heatmap instead of throwing and reading the existing snapshot.

## medium: Raw referrer values are written to server logs and analytics

id: fnd_sig-feat-library-30733ce202-36ba_874d2a2085
category: security
confidence: medium
triage: risk
status: open
feature: Node source src/pages (feat_library_30733ce202)

evidence:
- src/pages/llms.txt.ts:17-28 (GET)
- src/pages/llms.txt.ts:33-45 (GET)

The public endpoint copies the untrusted Referer header into a structured server log and into Vercel Analytics event properties. Referrer URLs commonly include paths and query strings, and can contain email addresses, one-time tokens, campaign identifiers, or other sensitive values from the linking page. Truncating the value limits size but does not remove sensitive query data, so requests to /llms.txt can persist private URL material in logs or analytics stores.

recommendation:
Parse the referrer and emit only a minimized value such as origin or hostname, or redact query strings/fragments before logging and tracking. Apply the same sanitization to console logs and analytics properties.

test analysis:
No tests are listed for this feature, and the code path directly handles request headers without a regression check for redaction or minimization.

suggested regression test:
Add a route test that sends a Referer containing a query token and verifies analytics/log payload construction only includes the sanitized origin or hostname.

minimum fix scope:
Sanitize the Referer header before the console log and track call in src/pages/llms.txt.ts.

## low: IntersectionObservers persist across Astro page loads without cleanup

id: fnd_sig-feat-library-f6b7911ff6-d59b_fdfccd3aa4
category: performance
confidence: medium
triage: risk
status: open
feature: Node source src/scripts (feat_library_f6b7911ff6)

evidence:
- src/scripts/scroll-reveal.js:15-26 (initScrollReveal)
- src/scripts/scroll-reveal.js:29

Each astro:page-load creates a new local IntersectionObserver and observes every .reveal element. The observer is only unregistered per element after that element intersects; elements that never intersect before an Astro navigation can remain observed, retaining detached DOM nodes and callback state across subsequent page loads. Over repeated client-side navigations this can accumulate observers and stale element references.

recommendation:
Track the observer and pending timers at module scope and disconnect/clear them on the relevant Astro navigation cleanup event before initializing a new observer, or reuse a single observer and skip elements already initialized.

test analysis:
No linked tests were included for this feature, so there is no coverage exercising repeated astro:page-load events with non-intersecting reveal elements.

suggested regression test:
In a DOM test, dispatch astro:page-load with a .reveal element that never intersects, then simulate a navigation cleanup/page-load cycle and assert the previous observer is disconnected or no stale element remains observed.

minimum fix scope:
src/scripts/scroll-reveal.js

## low: Malformed numeric XML entities can abort Substack feed parsing

id: fnd_sig-feat-library-f7615c5a6c-6478_fb8eddbd60
category: bug
confidence: medium
triage: risk
status: open
feature: Node package portfolio (feat_library_f7615c5a6c)

evidence:
- src/lib/data/substack.ts:40-49 (decodeXml)
- src/lib/data/substack.ts:77-99 (parseFeed)
- tests/lib/data/substack.test.ts:15-22 (stripHtml tests)

decodeXml converts any finite numeric entity with String.fromCodePoint, but it does not ensure the code point is within Unicode's valid range. An entity like &#9999999999; is finite but out of range, so String.fromCodePoint throws. Because parseFeed applies stripHtml/decodeXml while mapping items, one malformed entity can abort parsing the whole feed and force fetchEssays to fall back to stale or empty snapshot data.

recommendation:
Validate numeric entities with codePoint >= 0 && codePoint <= 0x10FFFF before calling String.fromCodePoint, returning the original entity for invalid values.

test analysis:
The Substack tests cover ordinary named entities, whitespace, read time, and a well-formed feed, but do not include invalid numeric entities or malformed feed content.

suggested regression test:
Add a stripHtml/decodeXml test with an out-of-range numeric entity and assert it does not throw and preserves or safely drops the invalid entity.

minimum fix scope:
src/lib/data/substack.ts decodeXml guard plus one Substack parser regression test.
